# PRD — Hero portraits on leaderboards / scatter / hero pool

**Status:** Proposed (V2.2)
**Owner:** unassigned
**Related:** [PR #20](https://github.com/jhiggins-tech/OW-Live-Report/pull/20); [V2 master plan](./v2.md).

## Context

PR #20 fetches OverFast's `/heroes` endpoint into `useHeroMeta()` and each
entry includes a `portrait` URL like
`https://d15f34w2p8l1cc.cloudfront.net/overwatch/<hash>.png`. None of V2's
chart surfaces use these yet — names render as plain text. Adding small
hero avatars where heroes are listed is a cheap visual polish that makes
each surface easier to scan.

## Goals

- Show a 24×24 hero portrait next to each hero name on:
  - The player page **hero leaderboard** rows.
  - The **team hero pool** bar chart's category axis (small avatar replaces
    or supplements the text label).
- Maintain existing fallbacks: if `useHeroMeta` errors or the hero isn't in
  the catalog, render the text label exactly as today.
- No extra network calls beyond image loading; data is already cached by
  PR #20.

## Non-goals

- Portraits on player scatter / roster cards / Overview stat cards. Those
  reference players, not heroes — the equivalent need (player avatars) is
  already covered by `player_summary.avatar` for the player profile header.
  Scoping this PRD to hero portraits only keeps it small.
- Lazy-loading or preloading hints. Browser default lazy loading is enough
  for ~30 small images.
- Storing portraits locally / hosting our own copies. Tekrop's CDN handles it.

## Proposed approach

1. **HeroLeaderboard** rows: add a leading column with a 24×24 `<img>`,
   `src={metaEntry?.portrait ?? ''}` with `loading="lazy"` and a
   transparent fallback.
2. **HeroPoolBar** (if we end up doing the meta overlay PRD): replace the
   plain text Y-axis label with a small `<img>` per category. Recharts'
   `tick` prop accepts a custom React component; render a 16×16 portrait
   centred on the tick.
3. Small CSS: `border-radius: 4px`, `box-shadow` faint, fixed size, alt
   text = hero name.
4. Network fault tolerance: empty `src` if missing portrait — let the
   browser show its broken-image placeholder OR render only the text label
   when there's no portrait URL.

## Open questions

- **Player scatter dot icons**: it's tempting to plot each player's dot as
  their hero portrait. But scatter dots correspond to players, not heroes;
  the natural icon is `player_summary.avatar`. Defer.
- **Hover state**: portrait could swap to a larger version on row hover.
  Skip for V2.2 — keep it static, ship faster.
- **`subrole` data**: OverFast returns a subrole (e.g. `tactician` for Ana,
  `sentinel` for Bastion). Could surface as a small badge near the
  portrait. Possible follow-up; not in this PRD.

## Risks

- Tekrop CDN going down (unlikely — it's been stable). With `loading="lazy"`
  + the existing text labels next to each portrait, a broken image is
  graceful: layout doesn't shift, name is still readable.
- Bundle size: zero. Images are loaded over HTTP on demand.
- Visual regression: portraits make the leaderboard taller. Audit row
  heights / line spacing.

## Acceptance criteria

- Hero leaderboard rows render with a portrait before the hero name.
- Heroes not in `HERO_CATALOG` (e.g. unknown new releases) still render the
  text fallback only.
- OverFast outage: leaderboard renders without portraits, layout intact.
- No new HTTP requests beyond image fetches on first render.

## Estimated effort

~½ PR, mostly markup. ~30–50 LOC in `HeroLeaderboard.tsx` + small CSS.
If shipped together with the meta-overlay PRD, the hero-pool bar piece
adds another ~30 LOC.
