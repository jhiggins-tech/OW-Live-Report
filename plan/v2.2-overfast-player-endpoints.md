# PRD — OverFast `/players/{id}/stats/*` endpoints

**Status:** Proposed (V2.2) — current recommendation **Skip** unless requirements change.
**Owner:** unassigned
**Related:** [PR #20](https://github.com/jhiggins-tech/OW-Live-Report/pull/20); [V2 master plan](./v2.md).

## Context

OverFast's OpenAPI exposes per-player endpoints beyond the `/heroes` family we
already use:

- `/players/{player_id}/stats`
- `/players/{player_id}/stats/summary`
- `/players/{player_id}/stats/career`

V1 uses these as its **primary** data provider: V1's `run-report.ps1` fetches
career stats from OverFast and writes them into the InfluxDB measurements V2
queries. So OverFast is upstream of InfluxDB in the V1 pipeline; InfluxDB has
**everything** OverFast does, plus history.

This PRD evaluates whether V2 should call these endpoints **directly** from
the browser, and lays out the trade-offs.

## Goals

- Document the trade-off cleanly so this conversation doesn't have to happen
  twice.
- Identify the one or two specific use cases (if any) where direct OverFast
  access offers something InfluxDB can't.

## Non-goals

- Actually shipping a direct-OverFast surface unless a clear win is identified.
- Replacing the InfluxDB query path. That stays the canonical source for V2.

## Analysis

### Data parity

| Capability | InfluxDB (today) | OverFast direct |
|---|---|---|
| Current rank per role | ✅ from `competitive_rank` | ✅ from `/stats/summary` |
| Current hero stats | ✅ from `career_stats_*` | ✅ from `/stats/career` |
| Profile (avatar, namecard, endorsement) | ✅ from `player_summary` (after PR #13) | ✅ from `/stats/summary` |
| **Historical time series** | ✅ that's what InfluxDB is for | ❌ snapshot only |
| **Freshness window** | last CI run (~hourly cap) | up-to-the-minute |
| Per-hero career stats with multi-mode breakdown | partial — `career_stats_*` indexed by gamemode | full — OverFast separates by quickplay / competitive / arcade |
| CORS | ✅ open | ✅ open (PR #20 verified) |

### Latency and cost

- InfluxDB: 4-statement multi-query takes ~3–6s cold on Overview, ~1s warm.
- OverFast: per-player endpoint typically ~1–2s, but **one fetch per player**.
  9 players → 9 sequential or parallel HTTP requests = ~3–8s wall time even
  with browser HTTP/2 multiplexing.
- Net: **OverFast doesn't beat InfluxDB on speed.** It does beat freshness.

### The freshness argument

The one capability OverFast offers that InfluxDB cannot is **up-to-the-minute
data**. InfluxDB only contains what V1's last CI run wrote. If a player
just finished a session 5 minutes ago, OverFast knows; InfluxDB doesn't.

Use cases where this matters:

- **Live "current rank" badge** on the player profile that reflects right
  now, not the last CI scrape. Possible win.
- **In-session climb visualisation**: showing rank changes since the start
  of the current session. Useful but niche.

Use cases where it doesn't matter:

- All chart surfaces — they show trends over days to weeks. The current
  hour vs four-hours-ago is invisible at that resolution.

### Decision

**Recommended action: do nothing for now.** The marginal freshness benefit
doesn't justify a parallel data path with its own error handling, caching,
and rate-limit semantics. If a clear feature requirement appears that
needs real-time data (e.g. "show me my live rank during a session"),
revisit this PRD.

## If we did ship something, the minimal scope

Smallest concrete feature that uses OverFast directly and adds value:

- A **"Live"** badge on the player profile that displays OverFast's
  current rank when it differs from InfluxDB's last-known rank, with a
  timestamp. Implementation:
  - New `useLivePlayerProfile(battleTag)` hook that fetches
    `/players/{battleTag}-encoded/stats/summary` lazily (only when the
    player page is opened, not on Overview).
  - Render only when the OverFast rank ordinal differs from InfluxDB's
    by at least one division.
  - Cache aggressively (5-minute staleTime) to avoid hammering OverFast
    on every navigation.

This is ~80 LOC and ~1 day of work, but adds a new external dependency
to a feature path that already works. Hold until needed.

## Open questions

- Is "live rank" actually a workflow you do? If the team checks their site
  before queueing to see "am I 5 mins of grind from Plat 1", that's the
  pitch.
- Does OverFast rate-limit anonymous browser clients? Tekrop publishes a
  rate limit; PR #20's `/heroes` fetch is fine but per-player hits would
  scale faster.

## Acceptance criteria (if we DO ship)

- New `useLivePlayerProfile(battleTag)` hook returns OverFast's current
  rank summary.
- Player profile header optionally shows a "Live" pill when OverFast and
  InfluxDB disagree by ≥1 division.
- 5-minute staleTime; degrades silently on OverFast outage.

## Estimated effort

If implementing: ~1 small PR. But this is currently **recommended skip**
unless concrete demand surfaces.
