# PRD — Hash URL migration (HashRouter → BrowserRouter + per-player stubs)

**Status:** Proposed (V2.2)
**Owner:** unassigned
**Related:** [V2 master plan](./v2.md).

## Context

V2 ships today with `react-router-dom`'s **HashRouter**, so every internal
URL has a `#` (`/v2/#/players/melrose`, `/v2/#/settings`, `/v2/#/optimizer`).
That works fine on GitHub Pages without server-side rewrites — every URL
hits `docs/v2/index.html`, the SPA reads `window.location.hash`, mounts the
correct route. It was a pragmatic choice at V2 launch because static Pages
can't do path-based SPA fallback out of the box.

Migrating to BrowserRouter would canonicalise the URLs as
`/v2/players/melrose/`, `/v2/settings/`, etc. — but doing so on a static
host requires emitting one **HTML stub** per public route at build time so
Pages has a file to serve at each path. This PRD captures the full plan
plus the trade-offs already discussed.

## Goals

- Replace HashRouter with BrowserRouter so canonical URLs have no `#`.
- Each route URL serves a stub `index.html` from Pages directly; SPA boots
  from there and continues to mount the route client-side.
- Unknown URLs (typo, removed player, future routes) render V2's own
  `NotFoundPage`, not the bare GitHub Pages 404.
- Existing hash bookmarks (`/v2/#/players/kie`) still land users on the
  correct route, via a small migration shim.

## Non-goals

- Server-side rendering, prerendering of page content, or Lighthouse-grade
  SSG. Stubs are skeleton-only — the SPA still fetches data at runtime.
- Per-player `<meta property="og:image">` from `player_summary.avatar`.
  That's a follow-up; first ship gets generic OG tags.
- Stub files for `/players/<slug>/<sub-route>` deep paths. Each player has
  exactly one stub at `/v2/players/<slug>/index.html`.

## Proposed approach

### Build pipeline

1. **New `scripts/build-stubs.mjs`** runs after `vite build`. Reads
   `docs/v2/index.html` (the canonical SPA shell Vite just emitted) and
   the freshly-built `docs/v2/data/roster.json`. For each route that needs
   a stub:
   - `/v2/players/<slug>/index.html` for every roster entry
   - `/v2/settings/index.html`
   - `/v2/optimizer/index.html`
   - `/v2/404.html` (the SPA fallback for unknown paths)

   Each stub is a copy of the main `index.html` with two edits:
   - `<title>` rewritten to the route-specific title (player display name,
     "Settings — Team Name", etc.).
   - Optional: a `<meta property="og:title">` and `<meta property="og:description">`
     filled in so Slack / Discord link previews differ per stub.

   The bundle URL inside each stub stays the same — all stubs share the one
   content-hashed JS file Vite emits.

2. **Stale-stub cleanup.** Before emitting, the script reads the existing
   `docs/v2/players/` directory and any extras (e.g. `settings/`,
   `optimizer/`). It removes any subdirectory whose slug isn't in the
   current roster + reserved-routes set. This handles roster removals.

3. **`prebuild` order.** The npm script chain becomes:
   ```
   build:roster && build:config && vite build && build:stubs
   ```

### Vite config

- `base: './'` → `base: '/v2/'`. The portable-relative-paths setup goes
  away. Every asset in every stub references `/v2/assets/index-<hash>.js`
  absolutely.
- Vite's `build.emptyOutDir: true` keeps wiping `docs/v2/` clean each
  build; the new `build:stubs` step regenerates the stub tree after.

### Router

- `main.tsx`: `createHashRouter` → `createBrowserRouter`, basename
  `/v2/`.
- Add a tiny **hash migration shim** that runs before `RouterProvider`
  mounts. If `window.location.hash` starts with `#/`, parse the segment
  and `history.replaceState` to the canonical path:

  ```ts
  if (window.location.hash.startsWith('#/')) {
    const target = window.location.hash.slice(1); // '/players/melrose'
    window.history.replaceState(null, '', '/v2' + target);
  }
  ```

  ~6 LOC. Old bookmarks land on the right route without flicker.

### 404 fallback

- `docs/v2/404.html` is itself a copy of the SPA shell. GitHub Pages
  serves it for any path under `docs/v2/` that doesn't match a file.
- The SPA boots, BrowserRouter sees a path that no route matches, and
  mounts `NotFoundPage`. The URL bar still shows whatever the user typed
  (404.html returns 404 but the SPA renders normally — same trick the
  React Router GitHub Pages template uses).
- One gotcha: 404.html is served with HTTP 404, so a deep-link
  share-preview bot might mark the URL as broken. Acceptable.

### Tests / verification

- `vite preview --base=/v2/` after build → `/v2/`, `/v2/players/kie/`,
  `/v2/settings/`, `/v2/optimizer/`, `/v2/players/__nope__/` all render
  correctly.
- `git status docs/` shows changes only under `docs/v2/`; check stub
  cleanup deletes removed-player directories.
- Visit `/v2/#/players/kie` post-deploy → URL bar replaces to
  `/v2/players/kie/`, page renders without flicker.

## Open questions

- **Reserved-routes set.** Today: `settings`, `optimizer`. Future routes
  (e.g. `/v2/about`) need to be added to `build-stubs.mjs`'s reserved list
  or the script needs to derive them from the router config. Derivation
  is more robust but couples the build script to the router; explicit
  list is simpler. Start explicit.
- **Slug collisions.** Two players with the same `slug` would overwrite
  each other's stub. `build-roster.mjs` already de-dups slugs (PR #9), so
  this is handled upstream, but the stub builder should still verify and
  fail loudly if it sees a collision.
- **Trailing slash convention.** GitHub Pages serves `/v2/players/melrose/`
  via `players/melrose/index.html`. Linking without the slash
  (`/v2/players/melrose`) issues a 301 redirect on Pages — usable but
  slower. Internal `<Link>` should always emit the slash.
- **PR rollout safety.** This change can hide V2 if anything goes wrong.
  Two options: feature-flag with a `VITE_USE_BROWSER_ROUTER=1` env (read
  via runtime-config.json) so a deploy can flip it on and revert quickly,
  or just ship and revert via PR. Probably overkill — Pages serves the
  prior content the moment a revert PR's commit-back lands.

## Risks (severity-ranked)

1. **Wrong Vite `base` → blank page.** If `base: '/v2/'` is correct but
   `docs/v2/index.html` ends up at the root, every asset 404s and the
   site shows a blank screen. Mitigate with a `pnpm preview --base=/v2/`
   smoke before pushing.
2. **Missing or stale stubs** → some routes 404 or others serve outdated
   roster entries. Mitigate with explicit stub-cleanup logic + a
   verification step in the workflow.
3. **CI commit-back grows.** `docs/v2/players/` now contains ~N
   directories. The Linux deploy workflow's `git add docs/v2` already
   handles it, but the commit-back becomes noisier on roster changes.
4. **Per-player title leaks.** If we add per-stub OG tags but the roster
   contains player display names with HTML-unsafe characters, the
   stub-emit step needs to HTML-escape. Low risk for this roster; still
   handle.

## Acceptance criteria

- `/v2/` resolves at the Overview without `#` in the URL bar.
- `/v2/players/melrose/`, `/v2/settings/`, `/v2/optimizer/` all serve
  matching content directly from Pages.
- `/v2/players/__notreal__/` renders the SPA's `NotFoundPage` (not the
  GitHub Pages 404).
- Visiting `/v2/#/players/kie` redirects to `/v2/players/kie/` in the URL
  bar on first paint.
- Removing a player from `tracked-battletags.txt` causes the next deploy
  to delete their stub.
- `vite preview --base=/v2/` mirrors production correctly.

## Estimated effort

~1 PR, but bigger than the V2.1 PRs: ~200–300 LOC across router setup,
new build script, workflow tweaks, and the migration shim. Plus careful
manual QA on the preview before merging.
